# Pokémon TCG AI Battle Working Note (1편): 규칙 기반에서 RL 파일럿까지

> **2026-07-02에 처음 작성하고, 2026-07-15까지의 구현과 실패 사례를 반영해 다시 썼다.**
> 이 글은 완성된 RL 해법을 소개하는 성공담이 아니다. RL을 제대로 알지 못한 채 시작해서 무엇을
> 잘못 짚었고, 시행착오를 거치며 다음 구현 순서를 어떻게 정했는지 담은 실전 가이드이자 오답노트다.

대회 링크:  
[The Pokémon Company - PTCG AI Battle Challenge Simulation](https://www.kaggle.com/competitions/pokemon-tcg-ai-battle)


## 1부. 문제부터 다시 정의하기

RL 기법을 고르기 전에 이 대회에서 무엇을 학습해야 하는지부터 정리할 필요가 있었다. 이 장에서는
게임을 선택지 순위 문제로 바꾸고, 덱과 파일럿을 분리한 뒤, 뒤에서 계속 사용할 최소한의 용어만
소개한다.

### 1. 먼저 결론: 이 문제에서 RL은 무엇을 해야 하는가

Pokémon TCG AI Battle에서는 시뮬레이터가 현재 고를 수 있는 행동을 목록으로 제공한다. 에이전트가
규칙을 해석해 모든 행동을 처음부터 만들어낼 필요는 없다. 해야 할 일은 현재 상황에서 **목록에 있는
행동의 우선순위를 정하는 것**이다.

$$
a_t \in A_t = \text{Options}(o_t)
$$

여기서 $o_t$는 에이전트에게 보이는 현재 상황이고, $A_t$는 그 순간 고를 수 있는 행동 목록이다.
모델은 목록 안의 각 선택지 $c_i\in A_t$에 점수를 매긴다.

$$
s_i=f_\theta(o_t,c_i), \qquad
\pi_\theta(i\mid o_t,A_t)=
\frac{\exp(s_i)}{\sum_{j\in A_t}\exp(s_j)}
$$

이렇게 보면 이 대회에서 RL을 적용하는 길도 조금 선명해진다.

```text
규칙 기반 파일럿과 리플레이를 기록한다
-> 실제 선택을 따라 하는 행동 복제 모델을 만든다
-> 중요한 판단과 어려운 상대에서 생기는 오차를 고친다
-> 규칙과 학습 모델을 섞은 안전한 파일럿을 만든다
-> 결과와 상태가치를 이용해 좋은 행동에 더 큰 가중치를 준다
-> 충분한 평가 기반이 생긴 뒤 오프라인 RL과 자기 대전을 시도한다
```

처음에는 이 순서를 거꾸로 생각했다. "게임이니까 RL", "RL이면 자기 대전"이라고 너무 빨리
연결했다. 실제로 먼저 필요했던 것은 믿을 만한 데이터, 상태를 표현하는 방법, 제대로 된 상대 파일럿,
홀드아웃, 패키지 검증이었다. 이 기반이 없으면 PPO든 Q-learning이든 복잡한 방법을 얹을수록
평가 오류만 더 커진다.


### 2. 덱, 파일럿, 필드, 평가 체계를 분리해야 한다

초기에는 에이전트 점수를 곧 덱의 강함으로 받아들였다. 가장 먼저 고쳐야 했던 생각이다.

| 구성 요소 | 답해야 하는 질문 |
|---|---|
| **덱** | 이 60장 구성은 어떤 승리 조건과 잠재력을 갖는가? |
| **파일럿** | 현재 상황에서 어떤 행동을 골라 그 잠재력을 얼마나 실현하는가? |
| **필드** | 지금 실제 래더에는 어떤 덱과 운영 방식이 얼마나 존재하는가? |
| **평가 체계** | 로컬 상대와 게임 수가 실제 강도를 판단하기에 충분한가? |

따라서 실제로 나온 성적은 대략 다음 요소가 함께 만든 결과다.

$$
\text{Observed Performance}
=f(\text{deck},\text{pilot},\text{opponents},\text{seat},\text{variance})
$$

같은 덱 해시라도 플레이 정책이 다를 수 있다. 반대로 같은 파일럿 코드라도 덱이 바뀌면 선택의 의미가
달라진다. 그래서 학습과 평가의 단위도 아키타입 이름 하나로 묶지 않고 다음처럼 더 잘게 나눴다.

```text
(date, team, exact_deck_hash, policy_or_epoch, opponent, seat)
```

메타 분석은 여전히 필요하다. 다만 이 글에서 메타는 주인공이 아니다. 메타의 역할은 다음 세 가지다.

1. 어떤 덱의 행동 데이터를 더 모아야 하는지 정한다.
2. 어떤 상대 파일럿이 벤치마크에 빠져 있는지 찾는다.
3. 현재 필드 비중에 맞춰 평가 상대의 비중을 다시 조정한다.

점유율 그래프와 일별 매치업은 Meta Snapshot 노트북에서 계속 다룬다. 이 글은 그 자료를 **RL
데이터셋과 평가 체계로 어떻게 바꾸는가**에 집중한다.


### 3. RL을 처음 보는 사람을 위한 최소 용어

#### 3.1 이 게임은 완전한 상태를 보여주지 않는다

상대 손패, 프라이즈, 덱에 남은 카드 전체는 보이지 않는다. 엄밀히 말하면 이 문제는 MDP보다
**부분 관측 마르코프 의사결정 과정(POMDP)**에 가깝다.

| 용어 | 이 프로젝트에서의 뜻 |
|---|---|
| 관측 $o_t$ | 현재 보드, 공개된 카드, 카운터, 로그, 이번 턴 사용 여부처럼 에이전트가 실제로 볼 수 있는 정보 |
| 선택지 목록 $A_t$ | 엔진이 현재 고를 수 있다고 제시한 가변 길이 행동 목록 |
| 정책 $\pi(a\mid o)$ | 관측 정보를 바탕으로 행동을 고르는 규칙 또는 모델 |
| 게임 궤적(trajectory) | 한 게임에서 이어지는 $(o_0,a_0,o_1,a_1,\ldots)$ 기록 |
| 보상 $r_t$ | 행동 뒤에 받는 학습 신호. 실제 목표는 최종 승패다. |
| 상태가치 $V(o)$ | 지금 상황에서 앞으로 얻을 것으로 기대하는 결과 |
| 행동가치 $Q(o,a)$ | 지금 특정 행동을 골랐을 때 기대하는 결과 |
| 어드밴티지 $A(o,a)$ | 그 행동이 현재 상황의 평균적인 선택보다 얼마나 나은지 나타내는 값 |

#### 3.2 행동 복제는 RL이 아니라 좋은 출발점이다

행동 복제(Behavior Cloning, BC)는 리플레이의 실제 선택을 정답으로 두는 지도학습이다.

$$
\mathcal L_{BC}
=-\log \pi_\theta(a_t^*\mid o_t,A_t)
$$

BC 자체를 RL이라고 부르기는 어렵다. 환경 보상을 직접 최적화하지 않고 기존 정책을 따라 하기
때문이다. 하지만 이 대회에서는 다음 이유로 매우 중요한 첫 단계다.

- 무작위 정책에서 카드 운용을 다시 배우지 않아도 된다.
- 수십만 개의 실제 판단을 지도학습 데이터로 쓸 수 있다.
- 선택지의 형태와 특징 추출이 제대로 연결됐는지 빠르게 확인할 수 있다.
- 이후 가치 학습, 오프라인 RL, 자기 대전을 시작할 초기 정책을 제공한다.

RL을 시작한다는 말은 곧바로 PPO를 돌린다는 뜻이 아니었다. 먼저 **학습 가능한 정책 표현과 믿을
만한 평가 체계**를 만드는 일이었다.

## 2부. 리플레이를 행동 복제 데이터로 바꾸기

문제를 정의했다면 다음은 학습 데이터다. 화려한 모델보다 먼저 상황과 행동을 정확히 연결하고,
학습과 실행이 같은 특징을 사용하도록 코드를 나눠야 한다. 이 장의 마지막에는 가장 단순한 행동 복제
파일럿을 실제로 실행할 수 있는 형태까지 만든다.

### 4. 리플레이에서 상황과 행동을 정확히 짝짓는다

CABT 리플레이를 확인해보니 step $i$에서 본 선택지에 대한 행동은 같은 칸이 아니라 다음 step의
`action`에 기록돼 있었다.

```text
features = steps[i][player].observation
label    = steps[i + 1][player].action
```

이를 잘못 맞추면 모델은 엉뚱한 선택을 정답으로 배운다. 정확도가 그럴듯하게 나올 수도 있어 더
위험하다. 파서는 반드시 다음 조건을 검사해야 한다.

```python
select = steps[i][player]["observation"].get("select")
action = steps[i + 1][player].get("action")

if not select or not select.get("option") or not action:
    return None

n_options = len(select["option"])
assert all(0 <= idx < n_options for idx in action)
```

첫 번째로 반환하는 60장 덱 리스트도 일반적인 게임 중 선택과 분리해야 한다. 길이가 60인 카드 ID
목록을 선택지 인덱스로 처리하면 데이터가 오염된다.

의사결정 하나를 저장할 때는 최소한 다음 정보가 필요하다.

```python
DecisionRecord = {
    "game_id": str,
    "date": str,
    "engine_version": str,
    "player": str,
    "team": str,
    "deck_hash": str,
    "opponent_deck_hash": str,
    "outcome": int,
    "turn": int,
    "context": str,
    "observation": dict,
    "options": list[dict],
    "chosen_indices": list[int],
}
```

저장할 때는 원본 JSON을 매번 다시 읽지 않고, 의사결정과 선택지 특징을 Parquet으로 변환한다.
날짜, 엔진 버전, 덱 해시를 남겨야 나중에 카드 풀이나 로그 형식이 바뀌었을 때 섞이지 않는다.


### 5. 코드 구조: 모델보다 각 단계의 역할을 먼저 나눈다

현재 가장 이해하기 쉬운 구조는 다음과 같다.

```text
Archive/MMDD/
  raw replay JSON

DecisionStore/YYYYMMDD/

PilotModelStore/YYYYMMDD/MODEL_ID/

  feature_schema.json
  train_manifest.json
  offline_metrics.csv

PilotStore/YYYYMMDD/PILOT_ID/
  main.py
  deck.csv
  model artifacts
  package_validation.json
  fidelity_report.json

pokemon_benchmark_runs/RUN_ID/
  benchmark_manifest.json
  matchup_matrix.csv
  runtime_report.json
  holdout_decision.json
```

코드도 이 흐름에 맞춰 역할별로 나누는 편이 좋다.

```text
리플레이 파서
-> 항상 같은 결과를 내는 특징 추출기
-> 데이터셋 생성기
-> 학습기
-> 실행 점수기
-> 안전장치를 포함한 에이전트 래퍼
-> 패키지 검사기
-> 매치업 / 재현도 평가 모듈
```

학습과 실행에서 특징을 따로 구현하면 거의 반드시 차이가 생긴다. 훈련에서는 쓰지만 실제 게임에서는
계산할 수 없는 특징이 들어가거나, 카드 ID와 상태 정규화가 달라질 수 있다. 따라서 하나의 특징 추출
모듈을 학습과 실행이 함께 사용하고, 같은 저장된 관측에서 완전히 같은 벡터가 나오는지 테스트해야 한다.

실행 시점에는 상대의 실제 아키타입, 덱 해시, 최종 승패, 다음 행동을 사용할 수 없다. 이런 값은
학습 데이터의 분류와 평가에는 쓸 수 있지만 모델 입력에는 넣으면 안 된다.


### 6. 0단계: 규칙 기반 정책의 판단 근거부터 기록한다

규칙 기반 파일럿은 학습 모델 이전의 임시방편이 아니다. 세 가지 역할을 한다.

1. 어느 정도 게임을 굴릴 수 있는 데이터 생성 정책
2. 학습 모델이 무엇을 바꿨는지 비교할 기준
3. 모델이 확신하지 못하거나 오류가 났을 때 사용할 안전장치

0단계의 에이전트는 행동만 반환해서는 안 된다. 왜 그 행동을 골랐는지도 함께 기록해야 한다.

```python
RuleDecision = {
    "chosen": [2],
    "rule_scores": [0.1, -0.5, 1.3, 0.2],
    "decision_family": "attack",
    "phase": "pressure",
    "fallback_used": False,
}
```

의사결정 유형은 특정 카드 이름보다 일반적인 기능으로 나눈다.

```text
setup / search / discard / attach / evolve / switch
attack / damage_target / effect_target / resource_management
```

이렇게 나눠두면 중요한 선택에 더 큰 학습 비중을 줄 수 있고, 모델이 어느 판단에서 자주 무너지는지도
찾기 쉬워진다.


### 7. 1단계: 고정 출력 분류기가 아니라 선택지 순위기를 만든다

행동 개수는 매 순간 달라진다. 따라서 클래스 0, 1, 2를 고정 출력하는 분류기는 적합하지 않다.
각 선택지에 같은 네트워크를 적용하는 순위기를 사용한다.

```python
class OptionRanker(nn.Module):
    def __init__(self, state_dim, option_dim, hidden=192):
        super().__init__()
        self.state_encoder = MLP(state_dim, hidden)
        self.option_encoder = MLP(option_dim, hidden)
        self.head = MLP(hidden * 2, 1)

    def forward(self, state, options, option_mask):
        # state: [B, state_dim]
        # options: [B, K, option_dim]
        h_state = self.state_encoder(state)[:, None, :]
        h_state = h_state.expand(-1, options.size(1), -1)
        h_option = self.option_encoder(options)
        scores = self.head(torch.cat([h_state, h_option], dim=-1)).squeeze(-1)
        return scores.masked_fill(~option_mask, float("-inf"))
```

이 구조는 선택지 표시 순서가 바뀌어도 같은 판단 기준을 유지하고, 선택지 수가 달라도 mask로 처리할
수 있다.

여러 장을 골라야 하는 상황에서는 고정 길이 출력을 한 번에 만들지 않는다. 하나를 고른 뒤 그 항목을
다음 계산에서 제외하고, `minCount`와 `maxCount`를 만족할 때까지 점수를 다시 계산한다. 학습
데이터도 같은 순서로 펼쳐 저장해야 학습할 때와 실제로 움직일 때의 처리 방식이 맞는다.

#### 상태 특징

- 현재 턴과 선공 여부
- 양쪽 프라이즈, 손패·덱·벤치 수
- 액티브와 벤치 포켓몬의 HP, 에너지, 도구, 상태 이상
- 이번 턴에 서포터·에너지·후퇴를 사용했는지
- 상대가 공개한 액티브, 벤치, 버린 카드와 에너지
- 최근 공개 행동을 요약한 정보

#### 선택지 특징

- 선택지 유형과 현재 판단 맥락(context)
- 참조하는 카드, 포켓몬, 공격 ID
- 대상 위치와 플레이어
- 공격 비용, 예상 피해, KO 가능 여부
- 진화 관계, 에너지 부착, 이동이나 교체 여부

카드 ID 임베딩만 쓰면 특정 60장을 외우기 쉽다. 카드 ID와 함께 HP, 타입, 진화 단계, 공격 비용,
공개된 능력 같은 구조적 특징을 넣어야 카드 한두 장이 바뀐 덱에도 조금 더 잘 대응한다.


### 8. 데이터셋을 어떻게 나누느냐가 모델보다 중요할 수 있다

#### 8.1 의사결정 행을 무작위로 나누면 안 된다

한 게임에서 나온 수십 개의 판단은 서로 강하게 연결돼 있다. 행 단위로 무작위 분할하면 거의 같은
게임 상태가 학습과 검증에 동시에 들어가 정확도가 부풀려진다.

최소한 게임 전체를 묶어 나누고, 가능하면 날짜와 정책 구간까지 분리한다.

```text
train      : 과거 날짜의 팀-덱-정책 구간
validation : 다른 날짜 또는 뒤쪽 정책 구간
holdout    : 후보를 확정한 뒤 처음 도착한 미래 날짜
```

팀이 하루 안에도 코드를 바꿀 수 있으므로 같은 팀과 덱 해시를 100게임 안팎의 연속 구간(epoch)으로
나눠 보는 방식도 사용했다. 이것이 실제 정책 변경 시점을 정확히 알아낸다는 뜻은 아니다. 서로 다른
운영 방식이 학습과 검증에 무작정 섞이는 일을 줄이는 장치다.

#### 8.2 승자 행동만 학습하면 생기는 문제

처음에는 이긴 쪽의 판단만 남기면 좋은 데이터가 될 것이라고 생각했다. 평균적인 라벨 품질은 올라갈 수
있지만 다음 편향이 생긴다.

- 좋은 매치업이나 운 덕분에 실수하고도 이긴 행동이 정답이 된다.
- 어려운 상대에게 잘 버티고도 진 게임의 좋은 판단이 사라진다.
- 특정 강한 팀과 덱의 행동이 과도하게 반복된다.
- 모델이 불리한 상태에서 회복하는 방법을 거의 보지 못한다.

현재는 양쪽 플레이어의 행동을 추출하고 다음 값을 별도로 보관한다.

```text
outcome
opponent strength
deck / team / epoch identity
decision family and impact weight
```

BC에서는 실제 선택을 그대로 정답으로 삼되, 결과와 상대 난이도는 샘플 비중을 조절하거나 이후 AWR
단계에서 활용한다.

#### 8.3 쉬운 선택이 중요한 선택을 덮지 않게 한다

선택지가 하나뿐인 판단은 실행 테스트에는 필요하지만 정책 학습에는 거의 정보를 주지 않는다. 반면 공격,
후퇴, 진화, 서치, 버리기, 공격 대상 지정은 빈도가 낮아도 한 번의 실수가 게임을 바꾼다.

따라서 다음 지표를 따로 낸다.

```text
overall top-1 / top-3
non-forced top-1 / top-3
critical-choice top-1 / top-3
decision-family accuracy
opponent and phase accuracy
mean chosen rank
```

결국 전체 정확도 하나만 높이는 것으로는 파일럿을 제대로 평가할 수 없었다.

## 3부. 행동 복제를 돌려보고 나서야 보인 문제들

첫 모델은 돌아갔고 오프라인 정확도도 나왔다. 문제는 그 숫자가 실제 게임 실력을 뜻하지 않았다는
점이다. 아래 네 가지 실패는 이후의 데이터 분할, 상대 구성, 홀드아웃 정책을 바꾼 직접적인 이유다.

### 9. 첫 번째 오답노트: 높은 모방 정확도는 강한 파일럿을 보장하지 않는다

가장 큰 착각은 오프라인 top-1 정확도가 높으면 파일럿도 강할 것이라고 믿은 것이다.

Dragapult의 특정 60장 덱은 로그에서 강한 상대를 만나고도 좋은 성적을 낸 기록이 있었다. 학습 파일럿도 홀드아웃
top-1 정확도가 60%를 넘었다. 하지만 전체 게임을 집중 검증하자 두 정책의 승률은 약 17.5%와
18.8%에 그쳤다.

왜 이런 일이 생겼을까?

1. 자주 나오는 쉬운 선택을 잘 맞히면 전체 정확도가 높아진다.
2. 초반의 중요한 순서 결정 한 번이 이후 상태를 완전히 바꾼다.
3. BC 정책은 작은 실수 뒤에 원래 파일럿의 기록에는 없던 상황으로 들어간다.
4. 실제 선택은 알려주지만, 선택하지 않은 행동이 얼마나 나빴는지는 로그에 없다.
5. 평균 행동은 비슷해도 특정 매치업의 승패 방향은 반대로 재현될 수 있다.

따라서 오프라인 정확도는 다음 질문에만 답한다.

> 기존 파일럿이 실제로 만났던 상황에서, 모델이 같은 선택을 얼마나 자주 다시 고르는가?

다음 질문에는 답하지 못한다.

> 모델이 자기 실수로 만들어낸 상태에서도 게임을 끝까지 잘 운영하는가?

이 간극이 행동 복제에서 말하는 분포 이동과 누적 오차 문제다. DAgger가 현재 정책이 실제로 찾아간
상황을 다시 모아 학습하는 이유도 여기에 있다.


### 10. 두 번째 오답노트: 아키타입 평균 정책은 특정 60장 덱의 운영법이 아니다

초기 순위기는 아키타입, 판단 맥락, 선택지 유형, 카드 ID의 빈도를 주로 사용했다. 구현은 간단하고
top-3 정확도도 그럴듯했지만, 보드 상황에 따라 같은 카드의 가치가 달라지는 문제를 해결하지 못했다.

```text
같은 공격이라도 상대 HP와 프라이즈 상황이 다르다.
같은 서치라도 손패, 덱, 버린 카드, 이번 턴 사용한 자원이 다르다.
같은 아키타입이라도 60장 구성과 승리 조건이 다를 수 있다.
```

그래서 v2에서는 **60장 덱 해시 + 현재 상태 + 각 선택지**를 함께 보는 쌍대 비교 순위기
(pairwise ranker)로 바꿨다. 실제로 선택한 행동의 점수가 나머지 행동보다 높아지도록 학습한다.

$$
\mathcal L_{pair}
=\log\left(1+\exp\left[-(s_{chosen}-s_{other})\right]\right)
$$

상태를 함께 보는 모델은 이전의 빈도 기반 모델보다 나았지만, 이것도 출발점일 뿐이었다.
2026-07-10 기준 특정 덱들의 오프라인 top-1은 약 58~70%까지 올라갔지만, 별도의 전체 게임
평가에서는 새 제출 후보가 기존 후보보다 낫다는 근거를 얻지 못했다.


### 11. 파일럿은 두 종류다: 필드를 재현할 것인가, 필드를 이길 것인가

한동안 모든 파일럿을 같은 기준으로 평가했다. 이것도 잘못이었다.

| 역할 | 목표 | 평가 기준 |
|---|---|---|
| **필드 재현용 파일럿** | 실제 래더의 해당 덱을 비슷하게 굴린다 | 행동 특성과 매치업 결과를 얼마나 잘 재현하는가 |
| **제출 후보 파일럿** | 같은 덱을 필드 평균보다 더 잘 굴린다 | 기준 파일럿 대비 우위, 새 홀드아웃, 실행 안정성 |

필드 재현용 파일럿은 벤치마크 상대다. 실제 필드에서 A 덱이 B 덱을 이기는 방향과 비슷한 결과를 내야 한다.
완벽한 모방에 가까울수록 좋다.

제출 후보는 다르다. 실제 필드보다 더 나은 결정을 내린다면 행동 모방 점수는 오히려 낮을 수 있다.
따라서 제출 후보에게 필드 모방을 강요하면 좋은 정책을 탈락시킬 수 있다.

이를 식으로 쓰면 목표가 다르다.

$$
\text{Field proxy}:\quad
\min_\pi |w_\pi(m)-w_{field}(m)|
$$

$$
\text{Adoption pilot}:\quad
\max_\pi w_\pi(m),
\qquad w_\pi-w_{reference}>0
$$

이제 학습 파일럿은 우선 필드 재현용 후보로 등록한다. 행동과 매치업을 잘 재현한다고 확인된 뒤
벤치마크 상대가 된다. 제출 파일럿은 별도의 탐색과 홀드아웃을 거친다.


### 12. 세 번째 오답노트: 약한 상대 파일럿은 후보를 강하게 보이게 한다

덱 리스트를 정확히 복사해도 상대 파일럿이 그 덱을 엉뚱하게 굴리면 벤치마크가 쉬워진다. 그러면
내 후보는 로컬에서 과대평가된다.

필드 재현용 파일럿은 최소한 두 가지를 점검한다.

#### 행동 재현도

- 주요 판단 맥락에서 선택 분포가 비슷한가?
- 공격, 서치, 후퇴, 진화 순서가 비슷한가?
- 초반, 전개, 압박, 마무리 단계의 행동 비중이 비슷한가?
- 드물지만 중요한 판단에서 모델이 대체 규칙에만 의존하지 않는가?

#### 상성 재현도

실제 필드의 매치업 승률 $r_m$과 로컬 재현 파일럿의 승률 $\hat r_m$을 비교한다.

$$
E_{fidelity}
=\sum_m q_m|\hat r_m-r_m|
$$

$q_m$은 현재 필드에서 그 매치업이 차지하는 중요도다. 평균 오차가 작아도 승패 방향이 뒤집힌
중요한 매치업이 있으면 벤치마크 상대로 쓰지 않는다.

2026-07-14에 만든 Festival 계열 두 재현 파일럿은 당일 자료와 비교했을 때 매치업 오차가 약
4.6~5.7%p로 작았다. 하지만 학습에 쓴 날짜와 평가 기준 날짜가 같았다. 그래서 검증 완료로 보지
않고 `needs_fidelity`로 남겼다. 다음 날짜에서도 같은 상성 방향이 나오는지 확인해야 한다.


### 13. 네 번째 오답노트: 같은 데이터로 고르고 검증하면 홀드아웃이 아니다

여러 모델과 규칙 혼합 비율을 시험한 뒤 가장 높은 결과를 고르면 그 값에는 운이 섞인다.

$$
\hat p_{selected}=\max_i(p_i+\epsilon_i)
$$

후보 수가 많을수록 그중 하나가 운 좋게 높게 나올 가능성도 커진다. 이런 선택 편향을 승자의 저주라고
볼 수 있다. 그래서 데이터마다 맡는 역할을 세 가지로 나눈다.

| 구간 | 용도 |
|---|---|
| 학습 / 탐색 | 특징, 모델, 혼합 비율을 고른다. |
| 날짜를 나눈 검증 | 과거의 다른 날짜에서 모델 계열을 비교한다. |
| 건드리지 않은 최종 홀드아웃 | 후보와 상대 패널을 모두 고정한 뒤 도착한 미래 날짜에서 한 번만 사용한다. |

2026-07-14 Spidops 실험에서는 새 후보가 55.77%, 기존 기준 파일럿이 54.81%였다. 차이는
+0.96%p로 사전에 정한 +3%p 기준을 넘지 못했다. 일부 가중 점수는 새 후보가 크게 앞섰지만,
표본이 작은 몇 매치업이 결과를 끌어올린 것이어서 승격하지 않았다. 이 홀드아웃은 이미 한 번 판단에
사용했으므로 이후 조정에는 다시 쓰지 않는다.

좋아 보이는 결과를 계속 들여다본 뒤 기준을 바꾸면 홀드아웃은 이름만 남는다. 후보, 상대 구성,
평가식, 통과선을 먼저 파일 해시와 실험 명세에 고정해야 한다.

## 4부. 학습 파일럿을 안전하게 실전에 연결하기

BC의 한계를 확인한 뒤에는 모델을 무작정 키우는 대신, 규칙 기반 정책과 역할을 나누고 상대에 대한
불확실성을 다루는 쪽으로 구조를 바꿨다. 목표는 학습 모델이 잘 아는 판단만 맡고, 나머지는 검증된
규칙이 받쳐주는 파일럿이다.

### 14. 안전한 실전 정책: 학습 모델과 규칙을 섞는다

현재 가장 현실적인 제출 구조는 신경망 단독보다 하이브리드다.

$$
s_{final}
=w_e s_{expert}
+w_g s_{global}
+w_r s_{rule}
$$

- `expert`: 특정 판단 유형, 게임 단계, 추정 상대에 맞춘 전문 모델
- `global`: 전체 데이터로 학습한 공통 순위기
- `rule`: 검증된 규칙 기반 점수

모델이 비슷한 판단을 충분히 학습했고 선택지 사이의 점수 차이도 클 때만 학습 모델의 비중을 높인다.
상대를 아직 파악하지 못했거나 해당 판단의 표본이 적거나 점수 차이가 작으면 규칙으로 돌아간다.

```python
if route_coverage < coverage_floor:
    return rule_choice(options)

scores = w_model * model_scores + w_rule * rule_scores

if confidence(scores) < confidence_floor:
    return rule_choice(options)

return legal_top_k(scores, min_count, max_count)
```

실행 래퍼는 다음 조건을 항상 지켜야 한다.

```text
첫 호출에서는 정확히 고정된 60장 덱을 반환한다.
게임 중에는 현재 선택지 안의 인덱스만 반환한다.
minCount와 maxCount를 지킨다.
중복 인덱스를 반환하지 않는다.
예외가 발생하면 엔진이 받아들일 수 있는 대체 행동을 반환한다.
로컬 저장소를 불러오지 않아도 독립 패키지에서 실행된다.
```

학습 결과가 아무리 좋아도 Validation Episode에서 패키지가 깨지면 소용없다. 모델 평가는 코드
불러오기, 초기 덱 반환, 첫 행동, 미러 게임, 실행 시간 검사까지 포함해야 한다.


### 15. 상대를 읽는 모델은 숨겨진 정답을 보면 안 된다

매치업별 전문 모델을 쓰려면 상대가 어떤 덱인지 추정해야 한다. 학습 데이터에는 실제 아키타입과 덱
해시가 있지만 실전에서는 알 수 없다. 실전 라우터가 사용할 수 있는 것은 공개된 카드뿐이다.

$$
P(z\mid x_{visible})
\propto P(z)\prod_{c\in x_{visible}}P(c\mid z)
$$

$z$는 상대 아키타입, $x_{visible}$은 액티브, 벤치, 버린 카드, 붙은 에너지처럼 현재 공개된
정보다. 초반에는 현재 필드 비중을 사전확률로 삼고, 카드가 공개될 때마다 아키타입별 확률을 갱신한다.

라우터는 다음 지표로 검증한다.

```text
turn별 top-1 / top-3 아키타입 정확도
turn별 예측 확신도와 엔트로피
판단 보류 또는 낮은 확신도의 비율
주요 아키타입 혼동 행렬
```

확신이 낮을 때 특정 상대용 전문 모델을 억지로 쓰지 않는다. 공통 모델과 규칙 기반 대체 행동으로
돌아가는 편이 안전하다.

공개된 카드 한 시점만으로 상대를 구분하기 어려우면 다음 단계에서 최근 공개 행동을 GRU나 작은
Transformer로 요약할 수 있다. 이때 hidden state는 게임 시작마다 초기화하고, 학습 배치도 게임의
시간 순서를 보존해야 한다. 단순한 피드포워드 모델보다 실제 상성 재현도와 확률 예측이 좋아졌을 때만
도입한다. 순환 모델이 숨겨진 카드를 알아내는 것은 아니다. 지금까지 공개된 정보로 상대에 대한 추정을
더 잘 요약할 뿐이다.


### 16. DAgger식 보강: 모델이 실제로 망가뜨린 상태를 배운다

BC의 가장 큰 문제는 기존 파일럿이 방문한 상황만 본다는 점이다. 모델이 실수하면 기록에 없던 상황으로
들어가고, 그곳에서 다시 실수한다. [DAgger](https://proceedings.mlr.press/v15/ross11a.html)는
현재 정책이 실제로 찾아간 상황을 모은 뒤, 그 상황에서 전문가가 무엇을 고를지 다시 물어 데이터에
보태는 방식이다.

이 대회에서 완전한 DAgger를 하기는 어렵다. 공식 로그의 원래 에이전트에게 새 상태의 정답을 다시
물을 수 없기 때문이다. 대신 다음과 같은 근사 방법을 쓸 수 있다.

```text
1. 현재 학습 파일럿으로 전체 게임을 돌린다.
2. 모델, 규칙, 여러 교사 정책이 서로 다른 행동을 고른 상황을 모은다.
3. 중요한 판단만 사람이 만든 규칙이나 검증된 교사 정책 묶음으로 다시 라벨링한다.
4. 여러 교사 정책이 합의한 인공 스트레스 게임의 판단만 제한적으로 학습한다.
5. 공식 리플레이에서 얻은 정답과 인공 게임에서 만든 정답을 구분해 보관한다.
```

교사 정책은 실제로 서로 달라야 한다. 같은 코드에서 상수만 조금 바꾼 변형은 별개의 전문가로 세지
않는다. 저장된 같은 상황에서 늘 같은 행동을 고른다면 실질적으로는 하나의 교사다.

인공 게임 궤적은 부족한 상황을 보충하는 데 유용하지만 다음 근거를 대신할 수는 없다.

- 실제 필드 파일럿의 행동 재현도
- 덱 파워 추정
- 날짜를 나눈 최종 검증
- 건드리지 않은 미래 홀드아웃

즉, DAgger식 보강은 학습할 상황을 넓혀주는 도구일 뿐, 실제 필드에서 검증한 증거를 대신하지는 못한다.

## 5부. 행동 복제 다음 단계로 넘어가기

여기서부터 비로소 보상과 가치 함수를 직접 다룬다. AWR, 오프라인 RL, 보상 설계, 자기 대전을 한꺼번에
시작하지 않고, 앞 단계에서 검증한 정책 표현과 평가 체계를 그대로 사용해 하나씩 확장한다.

### 17. 결과를 반영한 모방: BC 다음에는 AWR가 현실적이다

기본 BC는 모든 행동을 같은 비중으로 따라 한다. 다음 단계에서는 좋은 결과로 이어진 행동에 더 큰
가중치를 줄 수 있다.

이 단계부터는 BC 학습 기록을 RL에서 쓰는 상태 전이(transition) 형태로 확장해야 한다.

```python
Transition = {
    "observation": o_t,
    "options": A_t,
    "action": a_t,
    "reward": r_t,
    "next_observation": o_next,
    "next_options": A_next,
    "done": done,
}
```

여기서 `next`는 JSON의 바로 다음 행이 아니라 **같은 플레이어가 다시 판단하게 되는 시점의 관측**이어야
한다. 그 사이에 일어난 상대 행동과 엔진 처리 결과도 반영해야 한다. 게임 중 별도 보상이 없다면 중간
$r_t$는 0으로 두고 마지막 transition에 승패를 넣는 것이 가장 단순한 출발점이다.

$$
\mathcal L_{AWR}
=-w_t\log\pi_\theta(a_t\mid o_t,A_t),
\qquad
w_t=\min\left(w_{max},\exp\frac{\hat A_t}{\beta}\right)
$$

[Advantage-Weighted Regression](https://arxiv.org/abs/1910.00177)은 가치 함수를 먼저
추정한 뒤, 어드밴티지가 높다고 판단한 행동을 더 큰 비중으로 따라 하는 방식이다. 기존 BC 학습
구조를 거의 그대로 두면서 게임 결과를 반영할 수 있다는 점에서 이 프로젝트의 다음 단계로 알맞다.

다만 $\hat A_t$를 단순히 "이긴 게임이면 +1"로 두면 안 된다. 다음을 함께 고려해야 한다.

```text
최종 승패
상대 강도
매치업의 기본 유불리
현재 상태의 가치 추정
데이터 출처와 교사 정책의 신뢰도
```

가치 모델도 처음부터 복잡할 필요는 없다. 우선 최종 승패를 예측하는 보조 출력층으로 시작하고, 다른
날짜에서도 예측 확률과 실제 결과가 맞는지 확인한다. 가치 추정이 틀리면 나쁜 행동에 더 큰 비중을
줄 수 있으므로 기본 BC보다 오히려 나빠질 수 있다.


### 18. 오프라인 RL은 언제 시작할 것인가

리플레이에는 실제로 고른 행동 뒤의 결과만 있다. 고르지 않은 행동을 택했을 때의 결과는 없다. 일반적인
Q-learning을 그대로 적용하면 데이터에 거의 없는 행동의 가치를 터무니없이 높게 예측할 수 있다.

[Conservative Q-Learning](https://papers.nips.cc/paper/2020/hash/0d2b2061826a5df3221116a5085a6052-Abstract.html)은
데이터 분포 밖 행동의 가치를 보수적으로 다루는 방향이고,
[Implicit Q-Learning](https://arxiv.org/abs/2110.06169)은 데이터에 없는 행동을 직접 평가하지
않으면서 가치와 정책을 개선하려는 방법이다.

하지만 알고리즘 이름을 고르는 것보다 먼저 확인할 조건이 있다.

| 선행 조건 | 이유 |
|---|---|
| BC와 하이브리드 파일럿이 전체 게임을 정상적으로 끝낸다 | 기본 상태·행동 표현이 맞는지 확인해야 한다. |
| 날짜와 정책 구간이 분리된 데이터셋이 있다 | 오프라인 평가 누수를 막아야 한다. |
| 중요한 선택에 충분한 표본이 있다 | 쉬운 행동의 가치만 배우는 일을 막아야 한다. |
| 상대 재현 파일럿과 상성 방향이 검증됐다 | RL 개선 여부를 측정할 평가 체계가 필요하다. |
| 보상과 가치 예측의 신뢰도를 점검했다 | 잘못된 목표를 더 잘 최적화하는 일을 막아야 한다. |
| 아직 보지 않은 미래 날짜를 남겨뒀다 | 모델 선택 편향을 확인해야 한다. |

현재는 일부 덱에서 이 조건을 만드는 단계다. 따라서 CQL이나 IQL은 다음 실험 후보이지, 이미 검증된
해법이라고 쓰지 않는다.

또한 논문의 구현을 그대로 가져올 수 있는 것도 아니다. 이 게임에서는 고를 수 있는 행동이 매번
달라지므로 $Q(o,c_i)$를 현재 선택지마다 계산하고, 다음 상태의 Bellman target도
`next_options` 안에서만 만들어야 한다. 선택지 mask와 다중 선택 처리를 잘못 구현하면 보수적인
알고리즘을 쓰더라도 가치 함수가 잘못된다.


### 19. 보상 설계: 승리를 대신하는 지표를 목표로 만들지 않는다

가장 안전한 기본 보상은 최종 승패다.

$$
r_T\in\{-1,0,+1\}
$$

문제는 게임이 길어서 보상을 아주 늦게 받는다는 점이다. 중간 보상을 추가할 수는 있지만, 보조 지표가
실제 목표를 대신하기 시작하면 모델이 그 허점을 파고드는 보상 해킹(reward hacking)이 생긴다.

#### 위험한 예

- 드로우 수에 보상을 주면 필요 없는 드로우를 반복할 수 있다.
- 피해량에 보상을 주면 프라이즈 계획보다 당장 큰 피해를 우선할 수 있다.
- 벤치 전개에 보상을 주면 상대의 광역 공격에 취약한 보드를 만들 수 있다.
- 남은 덱 수를 무조건 높게 평가하면 필요한 서치도 피할 수 있다.

가능하면 잠재함수 $\Phi(o)$의 차이를 이용해 중간 보상을 만든다.

$$
r'_t=r_t+\gamma\Phi(o_{t+1})-\Phi(o_t)
$$

잠재함수 후보는 프라이즈 차이, 즉시 KO 위험, 다음 턴 공격 준비도, 자원 고갈 위험처럼 여러 요소를
작게 결합한다. 그래도 다음 검사를 통과해야 한다.

```text
shaping 없이도 최종 승률이 좋아지는가?
한 보조 지표만 비정상적으로 반복하지 않는가?
주요 매치업별로 효과 방향이 같은가?
규칙 기반 기준 정책보다 catastrophic behavior가 늘지 않는가?
```

처음에는 복잡한 중간 보상 없이 BC, AWR, 최종 결과 예측부터 확인하는 편이 안전하다.


### 20. 자기 대전은 마지막이 아니라 별도 리그로 설계한다

단일 덱끼리 반복하는 자기 대전은 그 덱의 거울전만 잘하는 정책을 만들 수 있다. 현재 메타에 없는
상대를 배우지 못하고, 두 정책이 서로의 약점에 적응하며 이상한 순환에 빠질 수도 있다.

자기 대전을 시작한다면 상대 풀을 한 정책으로 고정하지 않고 여러 정책이 함께 있는 리그로 만든다.

```text
현재 규칙 기반 기준 파일럿
검증된 필드 재현용 파일럿
최근 강한 공개 기준 에이전트
과거 정책 체크포인트
공격적 / 수비적 / 자원 고갈형 stress policy
현재 학습 정책의 여러 세대
```

상대 샘플링도 균등하게만 하지 않는다.

```text
현재 필드 비중
취약 매치업 우선순위
최근 패배율
정책 다양성
최근 평가가 뜸했던 상대를 다시 뽑는 비중
```

자기 대전의 승격 조건은 최신 자기 자신을 이겼다는 사실이 아니다. 고정해둔 과거 기준과 새
홀드아웃에서도 우위를 유지해야 한다. 다른 후보가 지는 상대를 이겨서 두 후보가 함께 무너지는 경우를
줄일 수 있는지도 봐야 한다.

## 6부. 알고리즘보다 먼저 갖춰야 할 평가 체계

학습 방식이 달라져도 평가 원칙은 바뀌지 않는다. 오프라인 정확도, 전체 게임, 상대 재현도, 다른 날짜,
미래 홀드아웃을 차례대로 통과해야 한다. 로컬과 라이브가 어긋났을 때도 먼저 원인을 나눈 뒤 고친다.

### 21. 단계별 검증: 앞 단계를 통과한 뒤 다음 단계로 간다

| 단계 | 확인할 것 | 통과하지 못하면 |
|---|---|---|
| 1. 데이터 연결 | 상황과 행동 정렬, 덱 반환 분리, 엔진 버전 | 파서부터 고친다. |
| 2. 오프라인 지표 | 강제 선택을 뺀 top-k, 중요한 판단, 다른 날짜 성능 | 특징과 데이터 비중을 고친다. |
| 3. 실제 실행 | 독립 실행, 60장 덱, 올바른 행동 형식, 미러 게임 | 패키지와 대체 규칙을 고친다. |
| 4. 행동 재현도 | 판단 유형·단계·상대별 행동 재현 | 필드 재현용으로 쓰지 않는다. |
| 5. 상성 재현도 | 실제 필드의 상성 방향과 오차 | 상대 구성을 고친다. |
| 6. 기준 대비 우위 | 같은 상대·좌석에서 기존 파일럿보다 나은가 | 제출 후보로 올리지 않는다. |
| 7. 날짜를 나눈 검증 | 다른 날짜에서도 효과가 유지되는가 | 과적합으로 기록한다. |
| 8. 미래 홀드아웃 | 고정된 후보가 처음 보는 미래 날짜에서 기준을 넘는가 | 새 홀드아웃 없이 다시 주장하지 않는다. |
| 9. 라이브 결과 대조 | 로컬 예측과 실제 결과의 차이를 설명할 수 있는가 | 평가 체계를 먼저 고친다. |

승률 하나만으로는 부족하다. 비슷한 강도의 상대끼리 매칭되는 래더에서는 강한 에이전트도 승률이
50% 부근에 머물 수 있다. 그래서 로컬에서는 모든 후보가 같은 상대를 만나게 하고, 선후공을 맞추며,
상대 강도와 매치업별 결과, 통계적 불확실성을 함께 본다.

공개 API에서는 엔진 난수를 완전히 고정할 수 없어 두 후보를 똑같은 랜덤 흐름에서 비교할 수 없었다.
대신 같은 상대 구성과 좌석 수를 맞추고, 충분한 게임 수와 Wilson 구간, 별도 홀드아웃을 사용해
우연의 영향을 줄였다.


### 22. 로컬에서는 좋고 라이브에서는 나쁠 때 무엇을 확인할까

"파일럿이 약해서"라고 바로 결론 내리면 안 된다. 원인을 나눠야 한다.

| 원인 | 진단 신호 | 다음 조치 |
|---|---|---|
| 상대 파일럿이 약함 | 로컬 매치업이 실제 필드보다 지나치게 쉬움 | 필드 재현용 파일럿을 다시 학습·검증한다. |
| 상대 구성이 빠짐 | 현재 점유율이 큰 덱의 로컬 게임이 없음 | 패널을 넓힌다. |
| 가중치가 오래됨 | 로컬 상대 비중과 현재 필드 비중이 다름 | 최신 자료로 다시 가중한다. |
| 제출 파일럿이 평균만 모방 | 행동 재현도는 높지만 기준 파일럿보다 낫지 않음 | 필드를 이기는 목적의 별도 탐색을 한다. |
| 승자의 저주 | 후보를 고른 뒤 새 평가에서 점수가 크게 하락 | 점수를 보수적으로 줄여 보고 새 홀드아웃을 쓴다. |
| 메타 변화 | 제출 뒤 상대 분포가 바뀜 | 날짜별 변화를 분리해 본다. |
| 실행 문제 | 에러, 대체 행동, 시간 초과가 라이브에서 증가 | 자기 대전 진단과 패키지 로그를 본다. |

핵심은 **무엇을 고치기 전에 원인을 측정하는 것**이다. 파일럿 문제가 아닌데 파일럿만 계속 튜닝하면
로컬 점수만 올라갈 뿐 라이브와의 차이는 그대로 남는다.

## 7부. 현재 위치와 다음 구현 순서

마지막 장은 지금까지 실제로 연결한 코드와 아직 하지 못한 일을 구분한다. 처음부터 다시 만든다면 어떤
순서로 진행할지, 같은 실수를 피하려면 무엇을 확인해야 하는지도 체크리스트로 남겼다.

### 23. 지금까지 실제로 구현한 범위

2026-07-15 현재 다음 흐름은 실제 코드로 연결돼 있다.

```text
공식 리플레이 파싱
-> 양쪽 플레이어의 의사결정 추출
-> 동일한 60장 덱별 선택지 데이터 생성
-> 상태를 함께 보는 쌍대 비교 순위기 학습
-> 독립 실행 가능한 소형 모델 패키지 생성
-> 초기 덱 / 첫 행동 / 미러 게임 기본 검사
-> 필드 재현용 파일럿의 상성 재현도 검사
-> 좌석 균형 벤치마크
-> 한 번만 쓰는 미래 홀드아웃과 결과 색인
```

최근 하루치 로그에서는 약 58만 건의 양쪽 플레이어 의사결정을 추출했다. 동일한 60장 덱별 학습 예시는
다음과 같다.

| 대상 | 의사결정 수 | 날짜·구간을 나눈 홀드아웃 top-1 | 현재 해석 |
|---|---:|---:|---|
| 새 Spidops 60장 덱 | 12,228 | 54.9% | 여러 날짜를 묶어 다시 학습해야 한다. |
| Festival 60장 덱 A | 19,372 | 71.8% | 당일 적합도는 좋지만 미래 날짜 검증 전이다. |
| Festival 60장 덱 B | 10,751 | 59.6% | 일부 매치업 오차가 커 추가 검증이 필요하다. |

이 숫자들은 학습 과정이 끊김 없이 돌아간다는 증거일 뿐, 제출 성능을 보장하지는 않는다. 실제로
Spidops의 새 후보는 미래 홀드아웃에서 기존 파일럿을 0.96%p 앞서는 데 그쳐 승격 기준을 넘지 못했다.

아직 완성하지 못한 부분도 많다. 다음 항목은 이제 시작했거나 앞으로 검증해야 한다.

- 여러 날짜를 합친 동일 60장 덱의 행동 학습
- 드물지만 중요한 판단과 취약 매치업의 상황 보강
- 교사 정책의 신뢰도가 다른 날짜에도 유지되는지 검증
- AWR에 쓸 가치 모델의 확률 예측 점검
- 오프라인 RL과 자기 대전의 독립 홀드아웃 평가
- 로컬과 라이브 사이의 장기 보정


### 24. 앞으로 코드를 작성할 순서

RL을 처음 접하는 입장에서 다시 시작한다면 다음 순서로 구현하겠다.

#### 구현 1단계: 상황과 행동 연결

```text
[ ] i 관측과 i+1 행동 정렬 테스트
[ ] 60장 덱 반환과 게임 중 행동 분리
[ ] 모든 정답 인덱스의 범위 검사
[ ] 엔진 버전, 날짜, 게임, 팀, 덱 해시 저장
[ ] 같은 게임이 학습과 홀드아웃에 동시에 들어가지 않음
```

#### 구현 2단계: 규칙 기반 기준과 판단 기록

```text
[ ] 판단 유형별 규칙 점수 저장
[ ] 대체 규칙 사용량 기록
[ ] 실행 시간과 에러 기록
[ ] 60장 덱 + 정책 실험 명세 생성
```

#### 구현 3단계: 행동 복제 v1

```text
[ ] 학습과 실행이 함께 쓰는 상태·선택지 특징 추출기
[ ] 가변 길이 선택지 순위기
[ ] 강제 선택 제외 / 중요 판단 / 판단 유형별 지표
[ ] 날짜 또는 정책 구간 단위 홀드아웃
[ ] 규칙 기반 대체 행동이 포함된 실행 래퍼
```

#### 구현 4단계: 필드 재현용 파일럿 검증

```text
[ ] 행동 특성 비교
[ ] 매치업 행렬 오차와 승패 방향이 뒤집힌 구간 검사
[ ] 현재 필드 커버리지 계산
[ ] 패키지 / 미러 게임 / 실행 시간 검사
```

#### 구현 5단계: 제출 파일럿 탐색

```text
[ ] 규칙과 모델의 혼합 비율을 작은 범위에서 탐색
[ ] 기준 파일럿과 같은 패널·좌석에서 비교
[ ] 선택용 게임과 최종 검증 게임 분리
[ ] 후보와 패널을 해시로 고정
[ ] 건드리지 않은 미래 홀드아웃을 한 번만 사용
```

#### 구현 6단계: 결과를 반영한 학습

```text
[ ] 확률 예측을 점검한 가치 출력층
[ ] 상대 강도와 최종 결과에 따른 가중치
[ ] AWR 또는 비슷한 어드밴티지 가중 BC
[ ] 기존 BC와 같은 단계별 검증 통과
```

#### 구현 7단계: 제한적인 오프라인 RL과 자기 대전

```text
[ ] CQL/IQL 소규모 실험
[ ] 학습 데이터에서 보기 힘든 행동의 가치 점검
[ ] 다양한 상대 리그 구성
[ ] 이전 체크포인트를 고정한 평가
[ ] 중간 보상을 제거한 비교 실험
[ ] 새 날짜 홀드아웃과 라이브 결과 대조
```

앞 단계가 실패했으면 다음 알고리즘으로 넘어가지 않는다. 모델 이름을 바꾸는 것이 파서, 파일럿,
평가 체계의 문제를 해결해주지는 않는다.


### 25. 다시 하지 않을 실수

| 잘못했던 접근 | 왜 문제였나 | 지금의 원칙 |
|---|---|---|
| 높은 라이브 점수의 덱을 복사하면 강할 것이라 생각함 | 덱과 파일럿을 분리하지 않았다. | 60장 덱과 실행 정책을 별도 산출물로 관리한다. |
| 전체 top-1 정확도를 성능으로 봄 | 쉬운 판단이 지표를 지배했다. | 중요한 판단·판단 유형·매치업과 전체 게임을 따로 본다. |
| 승자 행동만 정답으로 사용 | 생존자 편향과 매치업 편향이 생겼다. | 양쪽 행동을 보관하고 결과는 가중치로 사용한다. |
| 의사결정 행을 무작위 분할 | 한 게임 정보가 학습과 검증에 함께 들어갔다. | 게임·날짜·정책 구간 단위로 나눈다. |
| 같은 날짜에서 학습하고 재현됐다고 판단 | 다른 날짜로 옮겨갔을 때도 통하는지 알 수 없었다. | 다음 날짜를 별도 검증 자료로 남긴다. |
| 약한 상대 재현 파일럿으로 후보를 평가 | 로컬 승률이 부풀었다. | 행동과 상성 재현도를 먼저 검증한다. |
| 많은 후보 중 최고값을 그대로 믿음 | 승자의 저주를 무시했다. | 새 홀드아웃과 보수적인 수축 추정을 사용한다. |
| 단순 승률만으로 덱 파워를 판단 | 강한 상대를 만난 50%와 약한 상대의 60%를 구분하지 못했다. | 상대 난이도와 동일 덱·정책 구간을 함께 본다. |
| 처음부터 자기 대전을 계획 | 평가 체계와 상대 다양성이 준비되지 않았다. | BC와 하이브리드가 통과한 뒤 리그를 만든다. |
| 노트북 패키지 검사를 뒤로 미룸 | 모델이 좋아도 Validation Episode에서 실패했다. | 독립 실행과 미러 게임 검사를 학습 실험에 포함한다. |

이 표가 현재 프로젝트에서 가장 중요한 RL 결과일지도 모른다. 실패 원인을 분리하지 않으면 더 큰
모델이 같은 실수를 더 자신 있게 반복한다.


### 26. 참고 논문을 읽는 순서

처음부터 모든 수식을 이해할 필요는 없다. 현재 구현 단계에 맞춰 읽는 편이 낫다.

1. **행동 복제와 분포 이동**
   먼저 지도학습 정책이 왜 자기 실수 뒤의 상태에서 무너지는지 이해한다.

2. **[DAgger](https://proceedings.mlr.press/v15/ross11a.html)**
   현재 정책이 방문한 상태를 데이터에 다시 넣는 이유를 본다.

3. **[Advantage-Weighted Regression](https://arxiv.org/abs/1910.00177)**
   BC 구조를 유지하면서 좋은 행동에 더 큰 가중치를 주는 방법을 본다.

4. **[Conservative Q-Learning](https://papers.nips.cc/paper/2020/hash/0d2b2061826a5df3221116a5085a6052-Abstract.html)**
   오프라인 데이터에 없는 행동의 가치를 과대평가하는 문제를 본다.

5. **[Implicit Q-Learning](https://arxiv.org/abs/2110.06169)**
   데이터 분포 밖 행동을 직접 평가하지 않고 정책 개선을 시도하는 방법을 본다.

논문을 읽었다고 곧바로 전체 시스템을 바꾸지는 않는다. 지금 확인한 실패 원인 하나를 겨냥한 가장 작은
실험으로 옮기는 편이 안전하다.


### 27. 결론

이 프로젝트에서 RL은 규칙 기반 에이전트를 버리고 신경망으로 갈아타는 작업이 아니었다. 더 정확히는
다음 과정을 반복하는 일이었다.

```text
에이전트가 실제로 볼 수 있는 정보의 범위를 정한다
-> 실제 판단을 정확히 추출한다
-> 가변 길이 선택지를 순위화한다
-> 중요한 판단과 시간 이동을 따로 검증한다
-> 필드를 재현하는 상대와 필드를 이길 후보를 구분한다
-> 규칙 기반 대체 행동으로 실행 안정성을 지킨다
-> 결과 정보를 조금씩 반영한다
-> 새 홀드아웃에서도 우위가 남을 때만 다음 단계로 간다
```

지금까지 행동 복제와 상태를 함께 보는 순위기, 파일럿 패키지, 행동·상성 재현도 검사, 미래
홀드아웃까지는 실제로 연결했다. 반면 오프라인 RL과 자기 대전은 아직 준비 단계다. 어디까지 했고
어디부터 계획인지 분명히 적는 편이 중요하다.

가장 큰 교훈은 RL 알고리즘보다 먼저 **데이터와 평가의 역할을 나누는 것**이었다. 덱이 강한지,
파일럿이 강한지, 상대가 약한지, 같은 데이터를 반복해서 본 것인지 구분하지 못하면 어떤 점수도 쉽게
오해할 수 있다.

다음 Working Note에서는 여러 날짜의 동일 덱 행동을 합치는 방법, 중요한 판단의 표본을 보강하는
방법, 여러 교사 정책을 쓰는 방법, 날짜를 나눈 파일럿 검증을 실제 실험 단위로 더 자세히 다룰 예정이다.
